YoyoPod — Input Model

three-gesture contract · main button today, whole-surface touch tomorrow · single source of truth

Rev 2 — consolidation. Rev 1 assumed a touch panel with no hardware buttons. The Whisplay prototype has the opposite: exactly one physical button (the side button) and no touch — device/ui/src/hardware/whisplay.rs opens a single YOYOPOD_WHISPLAY_BUTTON_GPIO and returns one WhisplayButton; there is no separate PTT input. The gesture grammar below is unchanged — it is now defined independently of the input source. Decision log lives in README.md in this folder.

1 Gesture contract

Every navigation action is one of three gestures: press, double-press, or long-press. The grammar binds to whatever input source the hardware provides:

  • Today (Whisplay prototype): the single side button carries all three gestures. "PTT hold" is the same button, remapped per screen: on capture screens (Ask; the Record action in Talk) the runtime's passthrough policy turns a long hold into PttPress/PttRelease instead of long-press — exactly what OneButtonMachine::observe_ptt_passthrough (input/machine.rs) already implements. Consequence: while on a capture screen, hold = talk, not Home; leave via double-press navigation or the screen's own exit. There is no mid-hold cancel on one-button hardware — release always sends (acceptable: worst case is a two-second giggle note; discard afterwards from the Replay queue).
  • Future hardware: a dedicated PTT side button (hold-to-talk everywhere, long-press = Home restored on capture screens) and/or a touch panel where the whole surface acts as one big button carrying the same three gestures. Positional input is out of the contract — at 240 × 280 on a 1.69″ panel a 28 px row is a ~3.5 mm target, far below what a young child can aim at. The screen is never a precision surface. (Only exception: drag-off-to-cancel during walkie-talkie recording, touch binding only. See mockup_talk.html.)

The result: the whole device is operable with one motor skill, learnable in one sentence — press to look around, press-press to go in, hold to go home — and on a talking screen, hold to talk.

Two notes on consequences worth re-reading the table for:

  • The deck is always visible on Home and on every inner screen. Rev 1 hid it behind a long-press; that read clean but is not discoverable at age 4 — kids learn by seeing. The full-bleed "Blob only" composition survives as the Ambient state (§3): a presence state entered on idle timeout, not a navigation state.
  • Voice prompt on focus: every focus advance speaks the newly-focused item's label through the speaker ("Talk", "Mama", "Morning Songs"). This is the pre-reader bridge — list navigation never requires reading. Toggle lives in Setup → "Speak names".

2 Disambiguation timing

RangeClassified asNotes
0 – 180 mspressFires on release, not on press-down. Required so we can disambiguate from long-press. Applies identically to the hardware button and to a future touch surface.
180 – 400 msdead zoneReleased here = no action. Prevents accidental gestures from sloppy presses. The press-ring (see §4) does not appear in this range — only past 400 ms.
≥ 400 ms (still pressed)long-press armedVisual press-ring appears (§4.1). Releasing now (even at 401 ms) fires the long-press action.
Double-press window≤ 350 ms gap between two pressesTwo presses within 350 ms of each other count as a double-press. Any third press is ignored until the second press's processing completes. Open: this window needs testing with 4–6-year-olds; small hands may need it wider.

Runtime divergence — these numbers are the contract, not the shipped config. Today's classifier uses 800 ms long-hold and a 300 ms double window (input/config.rs: DEFAULT_LONG_HOLD_MS = 800, DEFAULT_DOUBLE_TAP_MS = 300; mirrored in config/device/hardware.yaml), and it has no dead zone — any release below long_hold_ms becomes a press candidate. Implementing this contract means: set 400/350 in the config, and add the 180–400 ms dead-zone rejection to OneButtonMachine. Tracked in HANDOFF.md → runtime work items.

3 State machine

Three top-level UI states. Home — Blob on stage, deck visible, press cycles deck focus. Ambient — Blob full-bleed, deck hidden: a presence state entered after the idle timeout on Home, exited by any input. Inside-a-slot — the normal application state; the deck stays visible and the screen's own container carries focus.

idle ≥ 30 s on Home ┌─────────────┐ ───────────────▶ ┌──────────────┐ │ HOME │ │ AMBIENT │ │ blob + deck │ ◀─────────────── │ blob only, │ └─────────────┘ any input │ deck hidden │ │ ▲ (focus kept) └──────────────┘ │ │ │ │ long-press (from anywhere inside) │ │ │ double-press on focused slot ▼ │ ┌─────────────────┐ │ INSIDE A SLOT │ press = advance in-screen focus │ (Listen, Talk, │ double-press = activate focused element │ Ask, Setup…) │ long-press = go Home └─────────────────┘ press on Home = advance deck focus (Listen → Talk → Ask → Setup → Listen …)

4 Visual feedback

4.1 — Long-press press-ring

Beginning at the 400 ms threshold, a 40 × 40 periwinkle ring fades in, centered in the stage (on button hardware there is no touch point; on future touch hardware it may follow the finger). It is a confirmation that the gesture has been recognized, not a progress indicator (there is no further threshold to cross). If the user releases at this moment, the long-press action fires. If they keep holding, nothing else happens — the gesture has already been committed.

4.2 — Ambient exit (wake to Home)

The deck fades/slides up from below over 200 ms with ease-out. The previously-focused slot is highlighted in its slot color (lime/peri/butter/coral). The stage above the deck does not dim — the deck is purely additive.

4.3 — Slot advance (press on Home)

Focus moves to the next slot. The previously-focused slot returns to cream, the new slot picks up its color. No animation — instant swap. (Animating focus would feel sluggish at the cycling rate this gesture invites.) The speaker says the slot's name ("Talk") if Speak names is on.

4.4 — Entering a slot (double-press)

The slot's root screen pushes in from below over 200 ms. The deck stays in place — only the stage content transitions.

4.5 — Returning to Home (long-press)

The stack pops to Home over the same 200 ms window. The last-focused slot is preserved, so Home comes back exactly as it was left.

6 LVGL implementation notes

ConceptLVGL mapping
Top-level panel lv_obj_t *panel with two children: stage and deck. Deck is visible by default; it hides only in the Ambient state (full-height stage).
Gesture recognizer One classifier fed by the ButtonDevice GPIO reader (see device/ui/src/hardware/whisplay.rs — a single button), emitting EV_PRESS, EV_DOUBLE_PRESS, EV_LONG_PRESS using the timing in §2 (press on release, 400 ms long-press threshold, 350 ms double window). On capture screens the classifier switches to PTT-passthrough mode (long hold → PttPress/PttRelease, as OneButtonMachine::observe_ptt_passthrough + router/routes.rs passthrough policies already do). The same classifier consumes lv_indev touch events if final hardware ships a panel — whole surface = one button, coordinates discarded.
NavStack Ambient is not a stack push — it's a presentation state on Home. Entering a slot via double-press performs NavigationPolicy::Stack push of the slot's root. Long-press pops the whole stack back to Home.
Slot focus persistence Stored as a single uint8_t last_slot_idx in the navigation context. Persists across deep navigation; on Home it decays to idle (no focus) after 8 s without input (see mockup_home.html).
Ambient enter/exit animation lv_anim_t on deck.y, 200 ms lv_anim_path_ease_out. The home stage's bottom rebinds to deck-h on exit so it shrinks beneath. (Could also overlay if shrinking the stage is too expensive; see open question.)
Press-ring A reusable lv_obj_t kept hidden in the panel root. On the long-press threshold event, it is shown centered in the stage with a 200 ms fade-in scale-up animation (touch binding may position it at the finger instead).

7 Worst-case input cost

From Home with Listen focused, reaching Setup takes:

press        → Talk focused
press        → Ask focused
press        → Setup focused
double-press → enter Setup
─────────
4 distinct gestures (3 presses + 1 double-press)
~ 1.4 s of press time if executed without thinking

From deep inside another category, add one long-press (→ Home) up front: 5 gestures. This is acceptable for 4 slots. If we add a 5th slot later (e.g. for a content-specific shortcut), worst case grows by one press per slot. Beyond that, cycling stops feeling cheap and we'd want to revisit.

8 Things still worth deciding

  • Press-ring colour on long-press inside a slot. Right now it's always periwinkle (the deck's signal colour). Should it instead inherit the focused slot's colour? Argument for: visual continuity. Against: now the same gesture has two different visual treatments, which means more state for the renderer.
  • Ambient timeout. 30 s is a placeholder. Long enough that it never interrupts active use, short enough that the shelf state is usually the Blob, not a stale menu.
  • Double-press window for small hands. 350 ms is an adult default. Test with 4–6-year-olds; may need 450–500 ms (at the cost of slower single-press dispatch).
  • Haptic. A 10 ms buzz on the 400 ms long-press threshold would teach the gesture faster than the press-ring alone. Not in scope here, but worth ear-marking.
