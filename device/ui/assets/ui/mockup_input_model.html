YoyoPod — Input Model

240 × 280 single-touch panel · gesture contract · single source of truth

1 Gesture contract

The panel is a single-touch surface with no hardware buttons. Every navigation action is one of three gestures: tap, double-tap, or long-press. There is no swipe and no drag (drag is reserved for the walkie-talkie cancel motion inside TalkContact; see mockup_talk.html).

SurfaceGestureResult
Home or any inner screen (nav hidden) long-press ≥ 400 ms nav bar slides up · focus = last-selected slot
Nav visible single tap advance focus to next slot, wrapping (Listen → Talk → Ask → Setup → Listen)
Nav visible double-tap enter focused slot's root menu · nav hides
Nav visible long-press ≥ 400 ms go to Home · nav hides · last-selected slot is preserved
Inner screen, row visible tap on row row action (push next screen, toggle state, etc. — defined per screen)

Two notes on consequences worth re-reading the table for:

  • Long-press is the only way to summon the nav bar, and once it's visible the only way to escape without entering a slot is another long-press (→ Home). This is intentional: it makes the gesture set learnable in one sentence ("hold to navigate, tap to cycle, double-tap to enter, hold again for home"). The cost is that revealing the nav by accident requires some deliberate action to clear.
  • Single tap on Home with nav hidden is a no-op. The screen does not wake from sleep via tap (sleep/wake is handled outside this contract). This keeps the home stage from being a tap-target for anything the child might be near.

3 State machine

Three top-level UI states. The nav bar is either hidden (in which case the deck region shows nothing — the stage extends to the bottom of the panel) or visible (deck visible, slot focus highlighted). Inside-a-slot is the normal application state.

4 Visual feedback

4.1 — Long-press press-ring

Beginning at the 400 ms threshold, a 40 × 40 periwinkle ring fades in at the touch point. It is a confirmation that the gesture has been recognized, not a progress indicator (there is no further threshold to cross). If the user releases at this moment, the long-press action fires. If they keep holding, nothing else happens — the gesture has already been committed.

4.2 — Nav bar reveal

The deck slides up from below over 200 ms with ease-out. The previously-focused slot is highlighted in its slot color (lime/peri/butter/coral). The stage above the deck does not dim — the deck is purely additive.

4.3 — Slot advance (tap on nav)

Focus moves to the next slot. The previously-focused slot returns to cream, the new slot picks up its color. No animation — instant swap. (Animating focus would feel sluggish at the cycling rate this gesture invites.)

4.4 — Entering a slot (double-tap)

Nav bar slides back down (200 ms), the slot's root screen pushes in from below over the same window. Total transition: 200 ms — they overlap, not stack.

4.5 — Returning to Home (long-press while nav visible)

Same as 4.4 but the destination is the home stage with no deck. The last-focused slot is preserved so that the next long-press reveals nav in the same state.

6 LVGL implementation notes

ConceptLVGL mapping
Top-level panel lv_obj_t *panel with two children: stage (full-height when nav hidden) and deck (hidden by default).
Gesture recognizer Single lv_indev_drv_t reader. Custom event hook on panel root that classifies into EV_TAP, EV_DOUBLE_TAP, EV_LONG_PRESS using the timing in §2. LVGL's built-in LV_EVENT_LONG_PRESSED can drive the 400 ms threshold; tap vs double-tap is handled by a 350 ms post-release timer.
NavStack Nav visibility is not a stack push — it's a transient overlay state on the current screen. Entering a slot via double-tap performs NavigationPolicy::Stack push of the slot's root. Long-press-while-nav-visible (→ Home) pops back to the root home screen.
Slot focus persistence Stored as a single uint8_t last_slot_idx in the navigation context. Persists across deep navigation; reset only by entering a different slot.
Deck reveal animation lv_anim_t on deck.y, 200 ms lv_anim_path_ease_out. The home stage's bottom rebinds to deck-h on reveal so it shrinks beneath. (Could also overlay if shrinking the stage is too expensive; see open question.)
Press-ring A reusable lv_obj_t kept hidden in the panel root. On the long-press threshold event, it is positioned to the touch coordinates and shown with a 200 ms fade-in scale-up animation.

7 Worst-case input cost

With nav hidden and Listen as last-selected, reaching Setup takes:

long-press → nav shows, Listen focused
tap        → Talk focused
tap        → Ask focused
tap        → Setup focused
double-tap → enter Setup
─────────
4 distinct gestures (1 long-press + 3 taps + 1 double-tap)
~ 1.6 s of finger time if executed without thinking

This is acceptable for 4 slots. If we add a 5th slot later (e.g. for a content-specific shortcut), worst case becomes 5 gestures. Beyond that, cycling stops feeling cheap and we'd want to revisit (e.g. direct-tap on a visible slot).

8 Things still worth deciding

  • Press-ring colour on long-press inside a slot. Right now it's always periwinkle (the nav-bar's signal colour). Should it instead inherit the focused slot's colour? Argument for: visual continuity. Against: now the same gesture has two different visual treatments, which means more state for the renderer.
  • Tap on the stage above the nav. When nav is visible, where does a tap on the stage area (not on the deck) go? Right now it falls through to whatever's there. I'd argue it should also count as "advance focus" — otherwise the user has to aim for the 52 px deck strip at the bottom, which on a 240 × 280 panel means a lot of dead screen.
  • Inner-screen row tap vs nav-cycle tap. When nav is visible over an inner screen (e.g. Talk contacts list), does a tap go to "advance focus" or to "select the row underneath"? Current rule says advance focus — the row is unreachable while nav is visible. Sounds right but worth confirming.
  • Haptic. A 10 ms buzz on the 400 ms long-press threshold would teach the gesture faster than the press-ring alone. Not in scope here, but worth ear-marking.
