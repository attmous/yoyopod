YoyoPod — Setup Flow · Implementation Spec

Setup is the grown-up corner of the device — the one surface where denser, text-forward UI is acceptable. It still obeys the three-gesture contract on the single hardware button (press = advance focus · double-press = open / activate · long-press ≥ 400 ms = Home; see mockup_input_model.html). No touch, no positional taps — every screen below is fully traversable by cycling focus. Card surfaces inside Setup inherit a pale coral tint (mirror of Talk v4's pale-peri card): the user reads "I'm in Setup" without a label. This spec replaces and extends the shipped runtime's Power screen (see §11).

Listen — lime
Talk — periwinkle
Ask — butter

1 · Setup root — six rows, scrollable

Depth 1, no tab strip. Six rows on the pale-coral card; the 170 px viewport shows four full rows (plus a clipped peek of the next — the peek is the scroll affordance, the coral nub confirms position). Press cycles rows top-to-bottom and wraps (FocusPolicy::Wrap); the list auto-scrolls to keep the focused row in view. Every row shows its current value in the trailing slot so a grown-up can audit the whole device state without entering a single sub-screen.

2 · Volume — block meter, live

Depth 2, tab strip SETUP / VOLUME. A horizontal meter of ten 16 × 22 px blocks. Press = +1 block, wrapping 10 → 1. The change applies immediately — each press plays a short "boop" at the new volume, so the feedback is the sound itself; the blocks are just confirmation for the grown-up. Double-press confirms and pops back to the root (root's trailing value updates). Long-press goes Home — the change is still saved, because there was never an uncommitted state.

3 · Companion — pick who lives on Home

Depth 2, tab strip SETUP / COMPANION. Five rows — one per companion (see mockup_companions.html for the full variant drawings). Each lead slot is the companion's body color, so the list is a palette of who's available. Trailing ✓ marks the active one. Double-press selects: the device pops all the way to Home, where the newly chosen companion waves — the payoff for the change is seeing it, not a confirmation dialog. Five rows overflow the 162 px tabbed viewport → scroll nub visible.

Selection behavior. No confirmation step: the change is trivially reversible and seeing the new friend wave is the confirmation.

4 · Contacts — read-only roster

Depth 2, tab strip SETUP / CONTACTS. A read-only list of the contacts with their assigned avatar colors — identical avatars to mockup_talk.html. This screen exists so Talk's empty-state pointer ("Ask a grown-up to add some in Setup") lands somewhere honest: the grown-up arrives here, sees the roster, and reads where editing actually happens. Double-press on a contact is a no-op — the row plays a gentle two-frame shake (below) instead of doing something surprising like starting a call from inside Setup.

Why read-only on-device. Adding a contact means typing a name and pairing an account — impossible on a one-button device and not something a 5-year-old should do anyway. 5 · Theme — light / dark / auto

Depth 2, tab strip SETUP / THEME. Three rows: Light, Dark, and Auto (follows bedtime hours — dark during the configured wind-down window). Lead slots are literal surface swatches: cream, near-black #1B1B1F, and a half-half split. Trailing ✓ on the active choice. Double-press applies instantly (the whole panel restyles under your thumb — that's the confirmation) and the ✓ moves; the screen stays put so the grown-up can compare and switch back.

6 · Speak names — in-place toggle, no sub-screen

Decision: this row toggles in place. A whole sub-screen for one boolean is a navigation tax; the root row already shows the value. Double-press on the row flips the trailing value with a 150 ms tick animation, and the device says "Speak names on" / "Speak names off" aloud — which is also a live demo of exactly what the setting does. This is the pre-reader accessibility feature: when On, every focus change speaks the focused item's name (rows, deck slots, playlist titles, contact names).

Volume leaf is lighter: chrome + 2 tabs + card + glyph + value + 10 blocks + hint ≈ 34.

10 · Input contract — per screen

11 · Runtime mapping — replacing Power

This spec requires a runtime change. The shipped router (device/ui/src/router/routes.rs) has UiScreen::Power as the 4th Hub target (HUB_SELECT), with POWER_SELECT = &[SelectionTarget::AdvanceFocus] and FocusPolicy::Wrap — a single info screen where select just cycles focus. The Setup tree replaces and extends it:

ItemToday (routes.rs)This spec
4th Hub targetPushScreen(UiScreen::Power)PushScreen(UiScreen::Setup)
RoutesPower (leaf)Setup + 5 children: SetupVolume · SetupCompanion · SetupContacts · SetupTheme · SetupAbout
Speak names—No route. Row 5 of SETUP_SELECT = EmitIntent(settings.speak_names_toggle); the per-row SelectionTarget array already supports mixing PushScreen and EmitIntent (Hub does the same shape).
Setup root select—[Push(Volume), Push(Companion), Push(Contacts), Push(Theme), EmitIntent(speak_names_toggle), Push(About)]
Volume select—Press is repurposed: emit settings.volume_step (wrap 10 → 1) instead of AdvanceFocus; double-press = pop. Needs a small router extension (press-emits-intent leaf, precedent: POWER_SELECT's AdvanceFocus is already a non-push select).
New intents—settings.volume_step · settings.companion_set · settings.theme_set · settings.speak_names_toggle
Focus policiesPower = WrapAll Setup screens = Wrap (single button can only go forward — Clamp would strand the focus at the last row).
Battery / charge displayPower screen bodyFolds into SetupAbout (§7). dirty_region_for(Power, Power) → None (full redraw) migrates to (SetupAbout, Power); SetupAbout also subscribes to the Network domain.
Persistence / navEphemeral · StackUnchanged: all Setup screens Ephemeral, NavigationPolicy::Stack. Call overlays still preempt Setup like any stack screen.
UiScreen::Power4th Hub target + routeRemoved (or kept as deprecated alias for one protocol version if the companion app references it).

Settings persistence lives runtime-side (survives reboot; synced to the companion app when online). The UI only emits intents and renders the snapshot — same one-way data flow as music and call state.

12 · Open questions for the next pass

  • Parent gate. Should entering Setup require a grown-up gesture — e.g. long-press 2 s on the Setup deck slot instead of the normal double-press? Argument against a gate: nothing in Setup is destructive or spendable — worst case a kid maxes the volume (cappable from the app), swaps the companion, or flips the theme, all trivially reversible and arguably a feature ("my device, my bunny"). Argument for: Speak names off silently removes the pre-reader's accessibility aid. Current lean: no gate in v1; revisit if telemetry shows accidental Speak-names flips.
  • Per-child profiles. If two kids share a device, companion + volume + speak-names would want profiles. Deliberately out of scope for v1 — the router and settings intents are per-device. Flagging so the protocol's settings domain leaves room for a profile id later.
  • Bedtime hours editing. Theme·Auto follows bedtime hours, but where are they edited — on-device or app-only? Current lean: app-only (time entry on one button is miserable); the Theme screen just names the dependency ("bedtime"). If app-less setups matter, a coarse on-device editor (hour blocks, same block-meter primitive as Volume) is the fallback design.
  • Max volume cap from the companion app. When the app caps volume at, say, 7: do blocks 8–10 disappear (meter is 7 long) or render locked (cream with a lock tint, wrap happens at 7)? Lean: render locked — a shrinking meter looks broken; visible-but-locked tells the grown-up the cap exists and where to change it.
