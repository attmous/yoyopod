YoyoPod — Setup Flow · Implementation Spec · v2 wheel + flat

Setup is the grown-up corner of the device, re-presented on the system-wide pattern: wheel navigation + fully flat chrome + neutral bezel. One hardware button, three gestures (press = roll the wheel one step · double-press = open / activate · long-press ≥ 400 ms = Home; see mockup_input_model.html), plus the side PTT which Setup never remaps. Every roll speaks the newly focused label (the Speak-names accessibility feature, §6). Setup's accent is c_coral; the whole stage is tinted full-bleed bg_stage_coral — the user reads "I'm in Setup" from the surface itself, no card, no label. The wheel's normative spec lives in mockup_listen.html §3; the values used here are restated in §8. This spec replaces and extends the shipped runtime's Power screen (§11).

1 · Setup root — wheel of six

Depth 1 under Home — no context label; a bare "n/6" counter sits top-right at (200,26) instead, because six items live behind a three-slot window. Press rolls the wheel one step (180 ms, wraps 6 → 1, new label spoken); double-press opens the focused entry — except Speak names, which toggles in place (§6). Each tile is variant A-with-value: a 56 × 56 flat glyph plate top-center (radius 10, bg_cream_2 or the item's own color — never c_coral, which is reserved for the focus fill), the name at Mont-18, and the current value as a Mont-12 ink_soft sub line — a grown-up can audit the whole device state just by rolling once around.

2 · Volume — flat block meter, live

Depth 2, context label VOLUME. A horizontal meter of ten 16 × 22 px flat blocks — filled = c_coral, empty = bg_cream_2, radius 4, zero borders. Press = +1, wrapping 10 → 1. The change applies immediately and each press plays a short "boop" at the new volume — the sound is the primary feedback, the blocks are confirmation for the grown-up. Double-press confirms & pops to the root (the Volume tile's sub updates). Long-press goes Home — the change is already applied, there was never uncommitted state. Nothing here needs a focus ring: the meter is the whole screen and press acts on it directly.

3 · Companion — wheel of five

Depth 2, context label COMPANION. Five entries, one per companion (variant drawings in mockup_companions.html). The plate is a flat 56 px circle in the companion's body color — the wheel is a palette of who's available. The active one carries a "✓ current" sub. Double-press selects: emit settings.companion_set, pop the whole stack to Home, and the new companion waves — the payoff is seeing it, not a dialog. Double-press on the already-active entry takes the same path — idempotent, never an error.

4 · Contacts — read-only wheel

Depth 2, context label CONTACTS. A read-only wheel of the roster with the same avatar colors as mockup_talk.html — the sub line names the color, doubling as a color-legend for the grown-up. This screen exists so Talk's empty-state pointer lands somewhere honest. Double-press is a no-op: the focused tile plays a gentle two-frame shake (±2 px, 120 ms) and the device speaks the footer hint aloud — "Grown-ups add contacts in the companion app". The visible hint is the short form (Mont-12 fits 200 px); the spoken form is the full sentence.

5 · Theme — wheel of three

Depth 2, context label THEME. Light, Dark, and Auto (follows bedtime hours — dark during the configured wind-down window). The plates are literal surface swatches as flat 56 px circles: cream, near-black #1B1B1F (a content swatch of the dark surface — not chrome ink), and a half-half split. "✓ current" sub on the active choice. Double-press applies instantly — the whole panel restyles under your thumb, that's the confirmation — the ✓ moves and the wheel stays put so the grown-up can compare and switch back.

6 · Speak names — in-place toggle, no sub-screen

Decision (carried from v1): this entry toggles in place. A whole sub-screen for one boolean is a navigation tax; the root tile already shows the value. Double-press on the focused Speak-names tile flips the sub line with a 150 ms tick — the old value slides up 6 px and fades, the new one slides in from below — and the device says "Speak names on" / "Speak names off" aloud, which is also a live demo of exactly what the setting does. The wheel never leaves the root; no route exists for this entry (§11). This is the pre-reader accessibility feature: when On, every wheel roll and deck focus change speaks the focused item's name (tiles, deck slots, playlist titles, contact names). Render: see §1, right frame — the "5/6" state is the toggle's home. Turning it Off is the last thing the device says until it's re-enabled.

7 · About — flat static page (the old Power screen, absorbed)

Depth 2, context label ABOUT. No wheel, no rows, no boxes — a battery glyph up top and bare, aligned label/value text lines on the coral-tinted stage (labels x 24, values x 130, y from 60 step 24). This absorbs everything the shipped runtime's Power screen displayed (battery, charge state) and adds firmware, network and device name. Press cycles a spoken read-through of the lines (no visual focus — nothing here is actionable, so nothing earns a focus ring); double-press is a no-op with the same gentle shake as Contacts; long-press goes Home.

10 · Input contract — per screen

One hardware button, three gestures. Long-press ≥ 400 ms is always Home, from every Setup screen, with no unsaved-state warnings — Setup never holds uncommitted state (Volume applies live; toggles apply on double-press). Focus rings appear only where fill + size can't carry focus — in Setup that is nowhere: the wheels and the meter carry focus by fill.

11 · Runtime mapping — replacing Power

This spec requires a runtime change. The shipped router (device/ui/src/router/routes.rs) has UiScreen::Power as the 4th Hub target (HUB_SELECT), with POWER_SELECT = &[SelectionTarget::AdvanceFocus] and FocusPolicy::Wrap — a single info screen where select just cycles focus. The Setup tree replaces and extends it:

12 · Open questions for the next pass

  • Parent gate. Should entering Setup require a grown-up gesture — e.g. long-press 2 s on the Setup deck slot instead of the normal double-press? Against a gate: nothing in Setup is destructive or spendable — worst case a kid maxes the volume (cappable from the app), swaps the companion, or flips the theme, all trivially reversible and arguably a feature ("my device, my bunny"). For: Speak names off silently removes the pre-reader's accessibility aid. Current lean: no gate in v1; revisit if telemetry shows accidental Speak-names flips.
  • Per-child profiles. If two kids share a device, companion + volume + speak-names would want profiles. Deliberately out of scope — the router and settings intents are per-device. Flagging so the protocol's settings domain leaves room for a profile id later.
  • Bedtime hours editing. Theme·Auto follows bedtime hours, but where are they edited — on-device or app-only? Current lean: app-only (time entry on one button is miserable); the Theme wheel just names the dependency ("bedtime"). If app-less setups matter, a coarse on-device editor (hour blocks, same flat block-meter primitive as Volume) is the fallback design.
  • Max volume cap from the companion app. When the app caps volume at, say, 7: do blocks 8–10 disappear (meter is 7 long) or render locked? Lean: render locked — flat version: capped blocks stay bg_cream_2 at 45% opacity (the same "inert" dimming the wheel peeks use) and the wrap happens at 7. A shrinking meter looks broken; visible-but-dimmed tells the grown-up the cap exists and where to change it.
