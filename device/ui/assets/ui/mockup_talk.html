YoyoPod — Talk Flow · v5 · Contact Wheel + De-boxed Calls

De-boxing pass. (1) The peri card + 28 px rows become a 3-slot contact wheel — the focused tile is the only box on screen. (2) The tab strip retires; depth ≥ 2 gets one 12 px uppercase context label, no box. (3) The card tint becomes a full-bleed stage tint (bg_stage_peri). (4) Call overlays lose the cream card and pill buttons entirely. (5) Recording is a full-bleed tomato state. PTT contract, stuck semantics, per-contact replay, and the route inventory are all unchanged from v4.

Wheel and Arc-Hero primitive geometry, animation, and FFI tables are normative in mockup_listen.html §3/§4/§7; this file specifies only Talk's variants and deltas.

1 · What changed v4 → v5

v4v5Why
Peri-tinted card + cream 28 px rows Contact wheel — 3 slots, step 80, focused center tile is the only boxed element Kills the card-in-card-in-row nesting. Big 56 px avatars and Mont-18 names read at arm's length for a 3-year-old; the wheel motion makes "press = next" physical.
Tab strip breadcrumbs at depth ≥ 2 One 12 px uppercase context label at (10,26), no box Tabs were chrome pretending to be navigation. The label answers "where am I" in one word; the deck answers "how do I leave".
Card surface tinted pale peri Full-bleed stage tint bg_stage_peri (y 22–228, no inset, no border) Same "you are inside Talk" signal with zero box chrome.
Call overlays: cream card, state pill chip, 26 px ctrl buttons De-boxed: bare peri stage, uppercase state label, 88 px avatar + pulse, Mont-24 name, 44×44 round buttons (mint / tomato) A call is the most stressful moment on the device — fewer shapes, bigger targets, one glance to the green button.
Empty state: card + hero circle + two-line text De-boxed: 56×56 peri "+" circle + spoken two-line hint on the bare stage Same content, no card. The voice channel carries pre-readers (§2c).
Unchanged: PTT hold-to-record contract (§3), stuck-send semantics — badge on the avatar + dot on the deck slot (§2b), per-contact replay queue (§4), route inventory (§7) All still true from v4; only their rendering changed.

Talk root is the contact wheel directly — no sub-menu, no context label (depth 1). Variant-C tiles: avatar circle, name, status badge. Press = roll one step (voice speaks the newly focused name), double-press = open the contact, long-press = Home. Peeks above/below are scaled 0.7 at 45% opacity with no chrome, sliding under the status bar and deck — the panel's overflow:hidden and the opaque bars produce the wheel look.

3 · TalkContact wheel + hold-to-record

Depth 2 — context label MAMA. Variant-A action tiles: Call · Hold to record · Replay. The PTT binding is unchanged from v4: on current hardware "hold" is the push-to-talk side button — hold PTT while the Record tile is focused to record, release to send, press the main button while holding to cancel. This matches the shipped runtime (router/routes.rs: PttPress/PttRelease voice-capture passthrough). On a future touch panel the same contract binds to holding the Record tile, with drag-off-tile as the cancel motion.

Hold-to-record flow (carried from v4 §7 — contract unchanged)

"Hold" below = PTT side button held (current hardware) or finger held on the Record tile (future touch). "Cancel" = main-button press while holding (button) or drag off the tile (touch).

                 hold begins (Record tile focused)
                          │
                          ▼
              ┌──────────────────────────┐
              │  stage flips tomato      │
              │  ● timer ticks (Mont-24) │
              │  voice meter animates    │
              │  deck dims to 55%        │
              └─────┬──────────┬─────────┘
       hold ends    │          │   cancel motion
                    ▼          ▼
        ┌──────────────┐   ┌──────────────┐
        │ SEND         │   │ DISCARD      │
        │ → 250 ms     │   │ no send      │
        │   whoosh fx  │   │ no toast     │
        │ → return to  │   └──────────────┘
        │   TalkContact│
        └──────────────┘

  60 s hard cap → auto-release (sends).
  < 0.4 s hold → tap, no recording. "Hold to record" tooltip flash on the Record tile.

The 2 s undo card on Home — which used to live here as part of the auto-play story — stays dropped along with auto-play. If the user mis-records, they can either cancel mid-hold (main-button press; drag-off on touch), or, after sending, open the contact's Replay queue and discard the just-sent note from there. Acceptable: the worst case is "kid sends a 2-second nonsense note and Mama hears a giggle." Not worth a Home-screen UI element to prevent.

10 · Open questions for the next pass

  • Notifications on Home. Still punted. When a note arrives while the user is on Home, how do they find out without an auto-play mechanism? Options: small badge on the Talk deck slot (consistent with the stuck-dot pattern), a transient toast on Home, a peri pulse on the deck. To be designed.
  • Stuck send retry policy. When does a failed send retry automatically? Suggest: on next successful network connect, silently retry once. If still fails, the contact's badge goes tomato "!" and the deck dot appears. The user enters the contact to retry manually from there.
  • Replay scoping. The Replay queue is per-contact. What happens to a contact's old notes — auto-delete after N days? Cap at N notes? Probably cap (~20 notes per contact, drop oldest). Storage on the device is small.
  • Contact ordering. The wheel shows Mama / Papa / Grandma in a fixed order. Recency-based (last interaction focused first) or pinned (Setup defines order)? Instinct: pinned. A 5-year-old doesn't want their contact wheel reshuffling.
  • Empty-state "+" animation. NEW — does the plus circle deserve a slow breathing animation (scale ~1.0 → 1.06, ≈2 s loop, echoing the Home blob)? Costs one animation slot, zero extra objects. To decide.
