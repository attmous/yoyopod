YoyoPod — Talk Flow · v4

Simplification pass. (1) Talk root is the contacts list directly — no intermediate "Contacts / Messages" branch. (2) Card background inside Talk inherits the slot's peri tint, so the user sees "where am I" without reading a label. (3) Messages archive removed entirely; replay lives per-contact. (4) Companion-on-focused-row reverted — focus = solid peri fill, like every other slot. (5) Blob is single-state (idle breathing only); the four-state proposal from v3 is dropped. Input model lives in its own doc: mockup_input_model.html.

Talk — periwinkle

1 · What changed from v3

v3v4Why
Talk root = 2 rows (Contacts, Messages) Talk root = contacts list directly (Mama, Papa, Grandma, Uncle, …) One less screen to traverse. Contacts is the only thing kids actually do here; everything else (replay, history) lives inside the contact.
Messages archive existed as its own route Dropped completely. Auto-play on Home (which itself is now dropped, see below) was the daily case; the archive was already de-emphasized. With auto-play gone, replay lives per-contact, which is where users think about it ("hear what Mama sent") not "open the mail app".
4-state Home blob (idle / speaking / reply-armed / listening) Single-state blob (idle breathing). The 4-state blob was tied to auto-play + AI invocation + walkie-talkie reply, all of which were collapsed into one contextual gesture. With AI invocation dropped and auto-play removed (since there's no Home gesture to invoke reply anymore), the blob has no extra states to communicate.
Companion-on-focused-row (lead slot = live blob) Reverted — focus = solid peri fill of the row. Three reasons: (a) tiny 14 × 14 sprite was hard to read at panel scale, (b) consistency with how focus reads on Listen / Ask / Setup, (c) it conflicted with the new tinted card surface — peri-on-peri-on-peri became visually noisy.
Card background = cream everywhere Card background inside Talk = pale peri (#E7E5F7). Reinforces "you are inside Talk" without a label. Rows shift cream so they pop on top. Focus row goes a step lifted peri (#BFBCEC) — still inside the peri family, still readable.
Stuck row at Talk root Stuck contacts visible inline with a coral badge on the avatar; coral dot on Talk deck slot. The user thinks "the note to Mama didn't go through" not "there is a thing in the stuck queue." Surface the failure on the person, not in a queue. Tap the contact → retry inside TalkContact.

Not changed: TalkContact (3 rows — Call, Hold to record, Replay), the walkie-talkie hold gesture, the call overlays (Incoming, Outgoing, In-call, Muted).

3 · TalkContact + walkie-talkie recording

This matches the shipped runtime (router/routes.rs: PttPress/PttRelease voice-capture passthrough). On a future touch panel the same contract binds to holding the Record row, with drag-off-row as the cancel motion. Tab strip returns at depth 2.

7 · Hold-to-record gesture (unchanged)

"Hold" below = PTT side button held (current hardware) or finger held on the Record row (future touch). "Cancel" = main-button press while holding (button) or drag off the row (touch).

                     hold begins (Record row focused)
                          │
                          ▼
              ┌──────────────────────┐
              │   row turns tomato   │
              │   timer ticks        │
              │   other rows dim     │
              │   deck dims          │
              └─────┬──────┬─────────┘
       hold ends    │      │   cancel motion
                    ▼      ▼
        ┌──────────────┐   ┌──────────────┐
        │ send         │   │ DISCARD      │
        │ → 250 ms     │   │ no send      │
        │   whoosh fx  │   │ no toast     │
        │ → return to  │   └──────────────┘
        │   TalkContact│
        └──────────────┘

  60 s hard cap → auto-release (sends).
  < 0.4 s hold → tap, no recording. Shows "hold to record" tooltip flash on the row.

The 2 s undo card on Home — which used to live here as part of the auto-play story — is dropped along with auto-play. If the user mis-records, they can either drag off mid-hold (cancel) or, after sending, open the contact's Replay queue and discard the just-sent note from there. Acceptable: the worst case is "kid sends a 2-second nonsense note and Mama hears a giggle." Not worth a Home-screen UI element to prevent.

8 · Open questions for the next pass

  • Notifications on Home. Punted to next iteration per your note. The question is: when a note arrives while the user is on Home, how do they find out without an auto-play mechanism? Options: small badge on the Talk deck slot (consistent with the stuck-dot pattern), a transient toast on Home, a peri pulse on the deck. To be designed.
  • Stuck send retry policy. When does a failed send retry automatically? Suggest: on next successful network connect, silently retry once. If still fails, the contact's avatar gets the tomato ring + "stuck" badge. The user enters the contact to retry manually from there.
  • Replay scoping. The Replay queue is per-contact. What happens to a contact's old notes — auto-delete after N days? Cap at N notes per contact? Probably cap (e.g., 20 notes per contact, drop oldest). Storage on the device is small.
  • Contact ordering. Right now I show Mama / Papa / Grandma / Uncle in a fixed order. Should it be recency-based (last interaction at top) or pinned (Setup defines order)? My instinct: pinned. A 5-year-old doesn't want their contact list reshuffling.
  • Empty state copy. Resolved in the consolidation pass: the text stays (it serves parents and early readers), and the pre-reader channel is voice — entering an empty Talk speaks "No contacts yet — ask a grown-up to add some!" This follows the voice-prompt-on-focus principle (README.md → "Pre-reader principles"); no flow may require reading to complete.
