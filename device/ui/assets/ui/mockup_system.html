YoyoPod — System Overlays · Loading + Error · Implementation Spec

The last two runtime routes with no design: UiScreen::Loading and UiScreen::Error. Both are overlays (NavigationPolicy::Overlay, Persistence::Singleton) — they float over whatever screen was active, never replace it, and there is exactly one instance of each. Small and friendly: a kid aged 3–7 sees a gentle pause or a butter "oops", never a hang and never a stack trace. Voice speaks everything important — pre-readers complete both flows without decoding a word. Frames below sit over a plain cream stage for clarity; frame 2 shows real overlay semantics with a dimmed wheel behind. Flat system: no UI borders, no drop shadows — fills + radius only. Input model lives in mockup_input_model.html.

3 · Behavior + input contract

Both overlays are FocusPolicy::None with NO_SELECT — there is nothing to focus and single press is consumed silently. The whole contract is double-press, long-press, and timers.

EventLoadingError · recoverableError · unrecoverable
Appears when any operation blocks the UI > 300 ms (debounce — quick ops never flash a spinner) an operation fails recoverably, or Loading exceeds 8 s the runtime marks the failure non-retryable (e.g. hardware, provisioning)
Single press no-op in every state — FocusPolicy::None, select target NO_SELECT. Consumed, no error tone, no flash.
Double-press no-op (reserved) retry now — cancels the 4 s timer, dismisses, re-issues the failed intent no-op
Long-press ≥ 400 ms → Home; pending operation cancelled → Home; retry abandoned → Home — the only exit
PTT press / release ignored while an overlay is active — the mic never goes hot behind a scrim.
Auto behavior > 2 s → speak wait line once · resolves → dismiss · > 8 s → escalate to Error after 4 s → auto-dismiss + retry once; a second failure re-enters Error and waits for input (no retry loop) none — stays until long-press
On dismiss overlay children hide, scrim drops, deck opa restores ≤ 100 ms — the underlying screen was never torn down (Persistence::Singleton, one reusable instance per overlay).

Diagnostics rule: the device never renders or speaks an error code, exception text, or stack fragment. On every Error entry the runtime writes one structured diagnostic line (code, source intent, retry count) to the device log for grown-ups and support — the screen shows only the friendly message above.

Spinner: two viable implementations

OptionCostVerdict
A — 8 dots + existing spinner fx8 objects, no new FFI, fx already wired to the loading sceneShip this. Matches the mockup exactly; invalidation is 8 tiny rects per tick.
B — canonical lv_spinner1 object, but needs new externs (lv_spinner_create + arc styling) and draws an arc, not dotsFallback if the dot fx ever misbehaves; visually a different (arc) spinner.

Object budget: Loading = 1 scrim + 8 dots + 1 label = 10; Error = 1 scrim + 1 badge + 1 glyph + 1 message = 4. Both sit on top of the live underlying screen and stay well inside the ~15-object overlay budget. Font externs: Mont-12 and Mont-18 are already linked in the FFI; this spec needs Mont-14 (labels/messages) and Mont-24 (badge "!") — both require new externs. Glyphs ship as A8 icon assets at runtime, not emoji.

7 · Runtime mapping

The route/controller contract in §3 is implemented by the retained-screen system-overlay controller and typed SystemIntent path. Retry, auto-retry, timeout escalation, and long-press dismissal clear the local overlay state immediately, while the runtime owns retry metadata and re-issues the retained failed intent. Raw diagnostic fields remain log-only. This spec closes README open question #7 ("Loading / Error overlay design — routes exist in the runtime, no mockup").

Runtime factWhere
UiScreen::Loading and UiScreen::Error are registered routesdevice/ui/src/router/routes.rs
Both are NavigationPolicy::Overlay — they float over the active screen, never replace itroutes.rs
Both are Persistence::Singleton — one reusable instance each; re-entry re-shows, never re-createsroutes.rs
Both are FocusPolicy::None with NO_SELECT — single press is a designed no-oproutes.rs
Loading scene already carries the spinner: stage: "overlay_center", fx: ["spinner"]; error scene is overlay_center with no fxdevice/ui/assets/scenes.ron

Implementation work is therefore: add the four sys_* roles (§6), re-theme the spinner fx dots to flat ink, add the Mont-14 / Mont-24 externs, and wire the §3 timers (300 ms debounce, 2 s speak, 8 s escalate, 4 s auto-retry) into the overlay controller.
