YoyoPod — Ask Flow · Implementation Spec · Flat

Ask is voice Q&A with the AI: hold the side push-to-talk (PTT) button, ask a question, let go, hear the answer. It is a single stateful surface — no tabs, no list, no sub-routes, nothing to focus. The screen exists only to show which phase of the conversation the device is in: IDLE → LISTENING → THINKING → ANSWERING → IDLE. The stage tints full-bleed butter (bg_stage_butter #FFF0CC) — the v1 pale-butter inner card is retired with the flat pass, same full-bleed move as Listen v2's lime and Talk v5's peri. Everything on this screen is also spoken aloud — text is for grown-ups, never a requirement. Input model lives in its own doc: mockup_input_model.html.

1 · Ask — four states

All four states share one locked geometry: the 60 × 60 hero tile sits at the same position in every state, only its contents swap. Kids track the tile, not the words. On entry to Ask the device speaks "Ask me anything!" — a pre-reader never needs the text below the hero.

3 · State machine + input contract

Ask has nothing to focus — no rows, no controls. Single press is a no-op by design (FocusPolicy::None). The whole surface is driven by the PTT button and voice-worker events; the main button only contributes double-press (stop) and long-press (Home). Flat note: the system's one surviving stroke — the transient focus ring — never appears in this flow, so ink here is text + glyphs only.

EventFrom IDLEFrom LISTENINGFrom THINKINGFrom ANSWERING
PTT press emit voice.ask_start → LISTENING — (already held) barge-in: drop pending answer, emit voice.ask_start → LISTENING barge-in: stop playback, emit voice.ask_start → LISTENING
PTT release — emit voice.ask_stop → THINKING — (guard: ignore stray release) —
Answer arrival (voice worker) stale — discard stale — discard (new question in flight) → ANSWERING, speech starts n/a
Answer end (voice worker) — — — → IDLE, hint swaps to "Ask me another!"
Single press no-op in every state — nothing to focus. UiScreen::Ask is FocusPolicy::None; the press is consumed and ignored (no error tone, no flash).
Double-press no-op (reserved) no-op (PTT is held; ignore) cancel pending question → IDLE stop answer → IDLE
Long-press ≥ 400 ms → Home emit voice.ask_stop first (mic must not stay hot), then → Home → Home, pending answer dropped stop playback, → Home

Runtime note — matches shipped router

This table matches device/ui/src/router/routes.rs as shipped: UiScreen::Ask carries ASK_PASSTHROUGH (PttPress → IntentTemplate::VoiceAskStart, PttRelease → IntentTemplate::VoiceAskStop), its select target is EmitIntent(VoiceAskStart), and its focus policy is FocusPolicy::None. Because select (double-press) emits VoiceAskStart rather than a dedicated stop intent, the "stop answer" behavior is realized in the voice worker: an ask_start received while speaking (and with no PTT held) is treated as barge-in-stop — playback halts, and if no speech follows within the short arm window the session resolves back to IDLE. The UI never needs a fourth intent.

4 · Kid safety / tone + offline variant

ConcernRuleNote
Content filtering All answers come from the cloud voice worker with kid-safe filtering applied server-side. Filtering policy is specced in the cloud voice-worker docs, not here — the device trusts what it is handed and renders/speaks it verbatim.
Answer length Max ~45 s of speech, with a gentle wrap-up. The worker ends long answers with a soft close ("…and that's the short answer! Ask me again if you're curious."), never a hard cut. Progress bar is sized to the whole clip including the wrap-up.
Tone Warm, short sentences, first person, spoken-first. Every on-screen string in this flow is also spoken. A pre-reader can complete the entire loop — enter Ask, hold, ask, listen — without decoding a single word of text.
Cloud unreachable ANSWERING is replaced by ASK · OFFLINE (frame below). Auto-return to IDLE after 4 s. Friendly, not alarming: the cloud is "sleeping", not "broken". Device speaks the same line it shows. No retry button — the kid just holds PTT and tries again.

Montserrat only — this page's CSS renders the panels with the "Montserrat", "Segoe UI", system-ui stack. The runtime has no emoji font: the hero glyphs (? 🔊 ☁ + zzz) and the deck's 🎧︎ 💬︎ ? ⚙ all map to A8 icon assets (deck set already ships; Ask adds icon_ask_q, icon_speaker, icon_cloud_zzz).

Object budget (worst case, LISTENING): 1 panel + 1 statusbar (~8 children) + 1 stage + 1 hero tile + 7 wave bars + 2 labels + 1 deck (4 slots + 4 glyphs) = ~29 objects. ANSWERING is ~25 (progress bar instead of bars). The flat pass dropped the AskCard and every border/shadow style — nothing casts a shadow, so nothing needs a sibling shadow rect and no shadow FFI is required; the deck's focused pill costs no extra object (the slot obj is the pill rect). Below Listen's ~36 worst case; comfortably inside LVGL limits at 240 × 280. All four states are one screen with widget visibility flags — never create/destroy on state change.

6 · Open questions for the next pass

  • "Hear it again". When an answer ends we snap to IDLE. Should the post-answer IDLE offer a replay affordance (e.g., double-press within 5 s = repeat the last answer)? Cheap to add — the clip is still in memory — but it gives double-press two meanings on one screen. Alternatively the kid just asks again. Decide after first playtest.
  • Hint list source. "Try: why is the sky blue?" — is the rotating example list static firmware strings, or server-fed (age-tuned, seasonal, localized)? Static ships v1; server-fed needs a fallback set anyway, so the static list is required work either way.
  • Whisper mode at night. Should ANSWERING respect a quiet-hours volume cap (e.g., after the bedtime hour set in Setup, answers play at whisper volume with the wrap-up shortened)? Belongs to the global volume policy doc, but Ask is the flow most likely to be used under a blanket at 9 pm — flagging it here.
