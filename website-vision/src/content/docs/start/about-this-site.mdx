*This page explains the site itself: what it is, what it is not, and how to
work with it.*

## Two sites, one honest split

The yoyopod repository carries **two** documentation sites, and the split is
deliberate:

| Site | Accent | What it documents |
| --- | --- | --- |
| `website/` — the as-built docs | coral (device tokens) | What is **actually implemented today**, code-verified, with honest gap ledgers. The engineering source of truth. |
| `website-vision/` — this site | amber & midnight (the [brand kit](/company/brand-kit/)) | The **target state**: the documentation a complete hardware + software product would ship with. Structure decided, content pending. |

Nothing on this site is a promise that a feature exists. When a vision page
and an as-built page disagree, the as-built page wins.

## Who this site is shaped for

Three audiences, three entry paths from the landing page:

- **Families** — parents (the buyers) and kids 7–14 (the users): setup,
  everyday use, safety, and privacy, written for people who will never open
  a terminal.
- **Builders** — engineers and contributors: the hardware platform, the
  software platform, and the developer guide.
- **Company** — anyone who wants the *why*: mission, principles,
  anti-positioning, roadmap.

Two cross-cutting sections serve all three: **User Stories** (persona
narratives that show what the product is for) and **Applications** (the
experiences themselves: Listen, Talk, Locate, the Parent App, Setup).

## Content status: how to read this site

The structure is the ideal; the content is placed wherever the repository
actually has it. Every page is in one of three states, visible both in the
sidebar badge and in the aside at the top of the page:

| State | Sidebar badge | Meaning |
| --- | --- | --- |
| **Filled** | *(no badge)* | Condensed from real, as-built documentation. Ends with a *Sources* note. |
| **Partially filled** | `Partial` | Some sections carry real content; the rest is outline marked *Placeholder* inline. |
| **Placeholder** | `Placeholder` | No as-built content exists yet — the outline is the target structure. |

The badge map **is** the gap list: every `Placeholder` badge marks a real
gap between the product vision and what exists today. Sibling pages share
a heading skeleton (all story pages have the same four headings, all app
pages the same five, and so on) so the site reads consistently regardless
of state.

## How a placeholder becomes a page

1. Answer the page's open questions (or move them to an issue).
2. Build the thing, document it in the canonical repo docs, and let the
   as-built site absorb it first.
3. Condense it here — replace the outline under each heading with prose,
   keep the headings, swap the caution for a *Sources* note.
4. Remove the sidebar badge in `astro.config.mjs`.
5. If the page states something as *currently true*, verify it against the
   as-built docs first — when the two sites disagree, the as-built site
   wins.

## Where the vocabulary comes from

The structure is grounded in the repository's real product language —
`docs/product/` and the root README: "the first device before a smartphone",
the V1 pillars (whitelist calls & voice messages, live-ish location,
music/audio, the parent app), the on-device screens (Hub, Listen, Talk,
Setup), and the anti-positioning list. The stubs echo that vocabulary so
this reads as *yoyopod's* future docs, not a generic template.
